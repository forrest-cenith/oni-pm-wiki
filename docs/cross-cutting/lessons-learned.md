# Lessons Learned & Institutional Knowledge

## Top Recurring Issues

### 1. Challenge Language Inherits FAR/RFI Framing
**Problem:** Government source material uses procurement language that doesn't fit OTA challenges.
**Fix:** Rewrite around prototype need and novel approaches every time.

### 2. Challenge Too Narrow
**Problem:** Challenge pre-selects a solution instead of describing the mission need.
**Fix:** Move detail into LOEs and keep public challenge broad.

### 3. "There Are Discrepancies" Without Actionable Detail
**Problem:** Government or ACC-RI flags issues but doesn't specify what needs to change.
**Fix:** Push for bulletized required changes or redlines before engaging vendor. This is a major recurring blocker.

### 4. Cancelled or Missed Coordination Calls
**Problem:** Key alignment meetings get cancelled, stalling progress.
**Fix:** Replace with a written action memo with named owners and deadlines.

### 5. Proposal/SOW/ROM Misalignment After Selection
**Problem:** Vendor's proposal doesn't match the government SOW or pricing is inconsistent.
**Fix:** Run a structured comparison and require a clean updated package.

### 6. Loose Status Tracking
**Problem:** Status updates are inconsistent, making it hard to know where things stand.
**Fix:** Use disciplined lifecycle tracker spanning pre-award through execution with clear stage definitions.

### 7. Invoice Packages Not Tied to CLINs
**Problem:** Vendor invoices don't map cleanly to contract line items.
**Fix:** Use a repeatable CLIN-based invoice worksheet for every billing event.

## Core Institutional Knowledge

### The Marketplace Challenge Is Not the Contract SOW
- Broad front end (challenge), specific back end (SOW/LOEs)
- Selection often triggers another documentation cycle rather than immediate award

### The PM's Most Important Value-Add Is Translation
1. Translating government need into OTA challenge language
2. Translating reviewer feedback into actionable vendor edits
3. Translating vendor billing into ACC-RI-compliant invoice output

### Selection ≠ Award ≠ Execution
Use precise status language at every stage:
- **Selected** — Vendor chosen from challenge submissions
- **Awarded** — Contract package approved and signed
- **Fully Executed** — All signatures complete, work can begin
- **Executing** — Work is underway

### Post-Selection Is a Mini Requirements Reconciliation
Expect multiple revision loops between vendor proposal, government SOW, and ACC-RI requirements. Plan for it.

## Lessons from Real Examples

### Drafting: DTRA/MOA Challenge Example
- Demonstrated translating government RFI material into ONI challenge structure
- Showed the importance of maintaining the four required challenge fields
- Incorporated review feedback to strengthen prototype/novel emphasis

### Vendor Coordination: PM UAS / Griffon Example
- Demonstrated the full post-selection document flow: notify vendor → request updated proposal and forms → send to ACC-RI → encounter vague discrepancy feedback → need more precise guidance
- Key lesson: ONI needs an exact prompt to give the vendor and should simultaneously clarify with the government what needs to change
- Cancelled SOW collaboration call was replaced with written action list

### Challenge vs. LOE Distinction
- Challenge is broad and public-facing
- LOEs are more specific and severable
- LOEs are what eventually flow into contractual agreements
